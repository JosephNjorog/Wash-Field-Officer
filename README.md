## Architecture

This is a real, persisted application â€” not a static mock. Every page reads from and writes to a Postgres database through Next.js Route Handlers under `app/api/*`. The Zustand store (`lib/store.ts`) is a thin client-side cache: it fetches all data on load via `loadAll()`, and every mutation (check-in, inspection submission, complaint update, officer edit, etc.) calls the API and merges the authoritative response back into state.

```
Browser (Zustand store) â‡„ Next.js Route Handlers (app/api/*) â‡„ Drizzle ORM â‡„ Neon Postgres
```

### Database schema (`lib/db/schema.ts`)

| Table | Purpose |
|---|---|
| `officers` | Field officer roster (name, region, contact, status, daily target) |
| `assets` | WASH infrastructure: water kiosks, boreholes, pipelines, sanitation blocks |
| `inspections` | Submitted inspection reports (water flow, condition, chlorine level, GPS, photos) |
| `complaints` | Customer complaints with full lifecycle (open â†’ assigned â†’ in-progress â†’ resolved) |
| `daily_summaries` | Per-officer daily activity rollups (sites visited, reports, distance) |
| `check_ins` | Audit log of site check-in events |
| `reports` | Audit trail of every generated/exported report (type, format, filters, who, when) |

Migrations live in `/drizzle` and are managed with `drizzle-kit`. Enums (`asset_status`, `complaint_status`, `officer_status`, etc.) are real Postgres enum types, not just TypeScript unions.

### Seed data

`data/seed.json` (generated by a deterministic script, not committed-from-scratch by hand) contains realistic Kenyan WASH data: 16 officers across 12 regions (Nairobi, Kisumu, Mombasa, Nakuru, Eldoret, Machakos, Kakamega, Garissa), 60 infrastructure assets, 225+ inspections spanning 28 days, 50 complaints, and 480 daily summary rows. `scripts/seed.ts` loads this into Postgres (`npm run db:seed`).

## Authentication & Roles

There's no real credential system (by design, for a prototype) â€” but routing is genuinely enforced:

- **`/login`** â€” pick a role: sign in as the Supervisor, or pick any of the 16 field officers.
- A signed-in session is stored in cookies (`fw_role`, `fw_officer_id`, `fw_name`) and read by **`middleware.ts`**.
- **Supervisors** can reach `/dashboard`, `/officers`, `/assets`, `/complaints`, `/reports`, `/settings` â€” and are redirected away from `/field`.
- **Field officers** can only reach `/field` â€” any attempt to hit a management route bounces them back.
- Unauthenticated visitors are redirected to `/login`.

## Pages & Features

### `/` â€” Marketing landing page
Public page describing the product: hero, live stats pulled from seed data, feature grid, "how it works" steps, and role-based CTAs into `/login`. Includes Open Graph/Twitter card metadata, JSON-LD structured data, and a generated OG image for link sharing.

### `/login`
Role picker (Supervisor vs. Field Officer with officer dropdown). Sets the session cookie and redirects to the right dashboard.

### `/dashboard` â€” Supervisor home
- 4 live stat cards (active officers, open complaints, inspections this week, infrastructure alerts).
- Interactive Leaflet map with **Asset View** / **Officer Activity View** toggle. Clicking any asset or officer marker (or a row elsewhere in the app) **flies the map to that location** and shows a detail callout.
- **Recent Activity Feed** â€” clicking an entry jumps straight to the relevant asset or complaint's full detail page (not just a summary popup).
- **Officer Performance** table â€” sortable by sites visited, reports, complaints handled, last check-in.
- **Recent Reports** audit trail widget.
- Polls every 25s and **toasts a notification** when a complaint is newly logged, reassigned, or changes status â€” so the supervisor sees field activity from other sessions without refreshing.
- Functional global search (officers, assets, complaints) and a notification bell that surfaces real overdue complaints and non-functional assets.

### `/officers` â€” Field Officer Management
- Roster table with live status badges, region, contact info.
- **Add Officer** (creates a real row via `POST /api/officers`).
- **Edit** any officer (name, region, contact, status, daily target) via `PATCH /api/officers/:id`.
- Click a row to open a detail sheet: assigned sites, recent inspections, assigned complaints, and a 7-day activity chart.

### `/assets` â€” Infrastructure Asset Registry
- Filterable table (type, status, region, inspection date range) split with a live map.
- Selecting an asset (table row or marker) flies the map to it and opens a detail panel: condition score, last 5 inspections, assigned officer.
- **Assign Inspection** â€” reassign the asset to a different officer (`PATCH /api/assets/:id`).
- **Log Issue** â€” file a new complaint against this asset directly from the registry (`POST /api/complaints`).

### `/complaints` â€” Complaint Management Queue
- Kanban board: Open â†’ Assigned â†’ In Progress â†’ Resolved, drag-and-drop powered by `@dnd-kit`, with SLA timers (red past 48h).
- Filter bar (category, officer, region, date).
- Click a card to open a detail drawer: full description, officer reassignment, status history timeline, resolution notes, and Mark In Progress / Resolved actions â€” all persisted immediately.

### `/reports` â€” Report Builder
- Filter by report type (Officer Performance, Infrastructure Status, Complaint Summary, Water Quality), date range, region, officer, asset type.
- Live chart previews (Recharts: bar/line/pie) recomputed from filtered live data.
- **Export PDF** and **Export CSV** are real â€” they generate and download an actual file built from the filtered dataset (via jsPDF/autotable and Blob downloads respectively).
- Every export is logged to the `reports` table and shown in a **Recent Reports** audit trail (also visible on the dashboard).

### `/settings`
Supervisor profile and system preferences (notification toggles, default site target).

### `/field` â€” Field Officer Mobile View
Rendered in a 390px phone-frame. Five tabs:
1. **Tasks** â€” assigned sites with check-in, plus a **List/Map toggle** to see sites on a Leaflet map.
2. **Report** â€” inspection form with **real GPS capture** (`navigator.geolocation`) and **real photo capture** (file/camera input with live thumbnail previews), water flow/condition radios, chlorine level, notes.
3. **Complaints** â€” view and update complaints assigned to the officer, plus **log a brand-new complaint** spotted in the field.
4. **Activity** â€” weekly stats, 7-day submissions chart, sync history.
5. **Profile** â€” officer info, the offline-mode toggle, and sign out.

Polls every 20s and **toasts a notification** when a new site or complaint is assigned to the officer by a supervisor in another session.

**Offline simulation**: toggling "Simulate Offline" (in the Profile tab) makes check-ins, inspection submissions, and complaint updates queue in `localStorage` instead of hitting the API immediately. Toggling back online replays the queue against the real backend and reconciles state via a fresh `loadAll()`.

## API Routes

All under `app/api/`, backed by Drizzle/Postgres:

| Route | Methods |
|---|---|
| `/api/officers` | `GET` list, `POST` create |
| `/api/officers/:id` | `PATCH` update |
| `/api/assets` | `GET` list |
| `/api/assets/:id` | `PATCH` update (reassign, status, condition) |
| `/api/inspections` | `GET` list (filterable by officer/asset), `POST` create |
| `/api/complaints` | `GET` list (filterable by status/officer), `POST` create |
| `/api/complaints/:id` | `PATCH` update (status, assignment, resolution note) |
| `/api/daily-summaries` | `GET` list |
| `/api/check-in` | `POST` record a check-in |
| `/api/reports` | `GET` list, `POST` log a generated report |

All write routes validate input with Zod and return 4xx with field-level errors on failure.

## Local Development


## PWA

The app is installable: `public/manifest.json` defines icons (192/512/maskable), theme color, and start URL; `next-pwa` generates a service worker (`public/sw.js`, gitignored as a build artifact) that caches static assets and enables offline app-shell loading. Icons were generated from a single brand SVG via ImageMagick.

## What's intentionally out of scope

- Real authentication (passwords, OAuth) â€” the role-picker cookie session is a deliberate prototype shortcut.
- Real file storage for inspection photos â€” photos are captured and previewed client-side (`URL.createObjectURL`) but not uploaded to blob storage; only the count is persisted.
- Push notifications â€” in-app toast notifications cover the "you've been assigned something" use case instead.
- Multi-tenant / multi-institution support â€” this models a single WASH institution's operations.
